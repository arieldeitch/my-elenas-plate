# Documentation Standards

## 1. מטרת התיעוד

התיעוד נועד לאפשר המשכיות אמינה בין:

- המשתמש
- GPT
- Claude Code
- Lovable
- GitHub
- Supabase
- כלי AI עתידיים

התיעוד אינו ארכיון של כל שיחה ואינו תחליף לקוד.

הוא צריך לשמר את המידע הנדרש כדי להבין:

- מה המוצר אמור לעשות
- כיצד הוא בנוי
- מה כבר בוצע
- אילו החלטות התקבלו
- מה נותר לעשות
- כיצד להמשיך בבטחה

---

## 4. מתי לעדכן מסמך

יש לעדכן מסמך כאשר:

- דרישה השתנתה
- Feature הושלם
- התגלה Gap מהותי
- התקבלה החלטה ארכיטקטונית
- נוצרה Migration משמעותית
- התווסף סיכון או Blocker
- השתנה סדר העדיפויות
- נוצרה נקודת עצירה
- נדרש Handover לכלי אחר

אין לעדכן מסמכים עבור שינוי קוסמטי חסר משמעות, אלא אם הוא משנה כלל עיצוב או Pattern.

---

## 5. מניעת סתירות

כאשר שינוי משפיע על כמה מסמכים:

1. לעדכן את מסמך מקור האמת.
2. לעדכן מסמכי מצב או Handover שמסתמכים עליו.
3. להסיר ניסוח ישן שסותר את השינוי.
4. להוסיף Decision כאשר יש חשיבות להסבר מדוע התקבלה הבחירה.

אין להשאיר שתי גרסאות פעילות של אותה דרישה.

כאשר מידע היסטורי חשוב, יש לסמן אותו כ:

- Superseded
- Deprecated
- Historical

---

## 6. סטנדרט ל-`project-status.md`

המסמך צריך לכלול:

- תאריך עדכון
- Branch ו-Commit כאשר ידועים
- שלב הפרויקט
- מה עובד
- מה חלקי
- מה חסר
- מצב Database ו-Supabase
- מצב בדיקות
- Blockers
- סיכונים
- הצעד הבא

אין לכתוב שמרכיב עובד אם הדבר לא נבדק או לא נתמך במידע קיים.

---

## 7. סטנדרט ל-`decisions.md`

כל החלטה צריכה לכלול:

- מזהה
- תאריך
- סטטוס
- הקשר
- החלטה
- נימוק
- חלופות שנשקלו, כאשר רלוונטי
- השלכות
- מסמכים או קבצים מושפעים

סטטוסים אפשריים:

- Proposed
- Accepted
- Superseded
- Rejected

אין לתעד החלטה טכנית טריוויאלית כ-ADR מלאה.

---

## 8. סטנדרט ל-`todo.md`

כל משימה צריכה להיות:

- ניתנת לביצוע
- בעלת תוצאה ברורה
- מסודרת לפי עדיפות
- מסומנת בסטטוס
- מקושרת לדרישה או Gap כאשר אפשר

עדיפויות מומלצות:

- P0 — חוסם שימוש או מסכן נתונים
- P1 — נדרש ל-MVP
- P2 — שיפור חשוב שאינו חוסם
- P3 — עתידי

סטטוסים מומלצים:

- Not Started
- In Progress
- Blocked
- Done
- Deferred

אין להשאיר משימות שהושלמו כפתוחות.

---

## 9. סטנדרט ל-`claude-context.md`

המסמך צריך להיות תפעולי וקצר יחסית.

הוא צריך לענות על:

- מהו הפרויקט
- מה מצב הקוד
- מה נעשה בסשן האחרון
- מה הצעד הראשון הבא
- אילו קבצים לקרוא
- אילו פקודות להריץ
- אילו סיכונים קיימים
- מה אסור לעשות

אין להעתיק אליו את כל מסמך הדרישות.

## 10. סטנדרט ל-`gpt-handover.md`

המסמך מיועד לדיון, תכנון והכוונה ברמה גבוהה.

הוא צריך לכלול:

- מטרת המוצר
- החלטות פעילות
- מצב נוכחי
- פערים פתוחים
- הנחיות עבודה עם המשתמש
- פרומפט המשך מומלץ
- שאלות עסקיות פתוחות, אם קיימות

אין להפוך אותו ל-Log טכני מלא.

---

## 11. תיעוד קוד

יש להוסיף Comments בקוד רק כאשר הם מסבירים:

- מדוע נבחר פתרון לא ברור
- Constraint עסקי
- Edge Case מהותי
- Workaround זמני
- סיכון עתידי

אין להוסיף Comments שמתארים שורה ברורה.

Public functions או Domain APIs מורכבים יכולים לכלול Documentation קצרה כאשר הדבר מועיל.

---

## 12. תיעוד Database

לכל Migration משמעותית יש לתעד:

- מטרת השינוי
- טבלאות ושדות מושפעים
- Constraints חדשים
- RLS policies
- Backfill
- השפעה על נתונים קיימים
- דרך אימות
- Rollback או Remediation במקרה הצורך

שם Migration צריך להיות תיאורי.

---

## 13. תיעוד בדיקות

כאשר מתווספת זרימה קריטית, יש לתעד:

- מה נבדק אוטומטית
- מה דורש בדיקה ידנית
- אילו Edge Cases מכוסים
- אילו מגבלות ידועות נשארו

אין להציג מספר Tests בלבד כתחליף להבנת הכיסוי.

---

## 14. תאריכים, גרסאות וקישורים

יש להשתמש בתאריך ISO:

`YYYY-MM-DD`

כאשר ידועים, יש לכלול:

- Branch
- Commit hash
- Migration name
- Pull Request
- Environment

אין להמציא מזהים או גרסאות שלא נבדקו.

---

## 15. שפה

התיעוד יכול להיות בעברית עם מונחים טכניים באנגלית.

יש לשמור על:

- Naming טכני עקבי
- שמות קבצים באנגלית
- שמות ישויות כפי שמופיעים בקוד
- הימנעות מתרגומים משתנים לאותו מושג
