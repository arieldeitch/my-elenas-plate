# Development Principles

## 1. מטרת העבודה

המטרה אינה לייצר את הקוד המרשים ביותר.

המטרה היא לייצר מוצר:

- נכון
- יציב
- ממוקד
- מודרני
- Scalable
- Maintainable
- קל לשימוש
- קל להמשך פיתוח
- מיושר לדרישות

---

## 2. AI First

בכל מקום שבו ניתן להעביר עבודה לכלי AI בצורה אמינה ובטוחה, יש לעשות זאת.

כלי ה-AI צריכים לבצע ככל האפשר:

- כתיבת קוד
- תיקון Bugs
- Refactoring נדרש
- בדיקות
- Migrations
- תיעוד
- Code Review
- Git Operations
- ניתוח שגיאות
- הכנת Prompts
- עדכון Context

המשתמש יבצע רק פעולות שאינן ניתנות לביצוע בשמו, כגון:

- Login
- יצירת Secret
- אישור הרשאה חיצונית
- החלטה עסקית מהותית
- פעולה בלתי הפיכה בעלת סיכון

---

## 3. עצמאות טכנית

כאשר קיימת החלטה טכנית סטנדרטית:

- יש לבחור פתרון לפי Best Practices
- יש להמשיך ללא עצירה
- אין לבקש אישור YES מיותר

כאשר קיימות כמה חלופות סבירות, יש להעדיף את הפתרון:

- המודרני יותר
- ה-Scalable יותר
- ה-Maintainable יותר
- העקבי יותר עם הקוד הקיים
- הבטוח יותר
- שמפחית עבודה ידנית

אין לבחור פתרון מורכב רק משום שהוא מתקדם יותר.

---

## 4. שמירה על Scope

יש להישאר מיושרים ל-MVP.

אין להוסיף:

- Feature חדש
- Refactor רחב
- Library חדשה
- Pattern חדש
- Agent חדש
- שינוי UI רחב

אלא אם הדבר נדרש ישירות למימוש הדרישה הנוכחית.

רעיון טוב שאינו נדרש כעת יתווסף ל-Future Enhancements.

---

## 5. Design Authority

יש לשמור על עקביות מלאה של המוצר.

לפני יצירת רכיב או Pattern חדש, יש לבדוק:

- האם קיים רכיב דומה?
- האם ניתן להרחיב רכיב קיים?
- האם העיצוב תואם למסכים אחרים?
- האם השמות עקביים?
- האם ה-Flow תואם לדפוסים שכבר נבחרו?

יש להעדיף מערכת אחידה על פני פתרון מקומי מבריק אך חריג.

---

## 6. Best Practices מובנות

כל עוד אין הרחבת Scope, יש לכלול:

- Favorite Foods
- Recent Foods
- Autocomplete
- Auto Save
- Offline Queue
- Safe Optimistic UI
- Loading States
- Empty States
- Error Handling
- Undo כאשר מעשי
- Debounce
- Mobile First
- Responsive Design
- Reusable Components
- Modular Architecture
- Accessible UI
- Database Constraints
- RLS
- Migrations
- Generated Types כאשר מתאים
- Tests לזרימות קריטיות

---

## 7. טיפול בתקלות

כאשר מתגלה תקלה:

1. להבין את הבעיה.
2. לבדוק Logs ושגיאות.
3. לזהות Root Cause.
4. לנסות פתרון ממוקד.
5. להריץ בדיקות.
6. לבדוק Regression.
7. לעדכן תיעוד כאשר נדרש.

יש לערב את המשתמש רק כאשר:

- נדרשת גישה שאינה זמינה
- נדרש Secret
- קיימת החלטה עסקית
- קיימת פעולה מסוכנת
- מוצו פתרונות מקצועיים סבירים

---

## 8. איכות קוד

יש להעדיף:

- Type Safety
- פונקציות קטנות וברורות
- Naming עקבי
- רכיבים ממוקדים
- Queries ממוקדות
- Validation מפורש
- Error Handling צפוי
- קוד שניתן לבדיקה

יש להימנע מ:

- `any` ללא הצדקה
- קוד משוכפל
- Magic Values
- Components גדולים מדי
- Side Effects לא מבוקרים
- Queries רחבות
- State כפול
- קוד מת ללא צורך

---

## 9. Security

יש להקפיד על:

- Supabase RLS
- Secrets מחוץ ל-Git
- שימוש נכון ב-Environment Variables
- אי-חשיפת Service Role
- Validation בצד השרת או Database
- Sanitization כאשר רלוונטי
- מינימום הרשאות
- אי-רישום מידע רגיש ב-Logs

---

## 10. Database Changes

כל שינוי Database יתבצע באמצעות Migration.

Migration צריכה:

- להיות ניתנת להרצה מחדש כאשר מתאים
- לכלול Constraints
- לכלול Indexes נדרשים
- לכלול RLS Policies
- להיבדק מול נתונים קיימים
- להיות מתועדת

אין לערוך Production ידנית ללא תיעוד.

---

## 11. Git

יש לבצע Commits ממוקדים וברורים.

Commit צריך:

- לתאר שינוי אחד לוגי
- לכלול קוד ותיעוד רלוונטי
- לא לכלול Secrets
- לא לכלול קבצים זמניים
- להשאיר Build תקין

יש להימנע מ-Commit Messages כלליים כגון:

- update
- fixes
- changes

---

## 12. Quality Gate

לפני השלמת משימה יש לבדוק:

- Build
- Type Check
- Lint
- Tests רלוונטיים
- Imports
- Mobile Layout
- Error States
- Loading States
- Empty States
- Supabase Sync
- Realtime Behavior
- Scope Alignment
- Documentation

---

## 13. Product Thinking

לפני מימוש שינוי יש לבדוק:

- האם הוא פותר צורך אמיתי?
- האם הוא מפחית חיכוך?
- האם הוא מקל על התיעוד?
- האם הוא שומר על המיקוד באפליקציית תזונה?
- האם הוא מוסיף עומס?
- האם הוא נדרש ל-MVP?
- האם הוא משפיע על שני המשתמשים באופן עקבי?

---

## 14. מקור האמת

ההנחיה האחרונה של המשתמש גוברת על מידע קודם.

כאשר היא משנה דרישה:

- יש לעדכן את מסמך הדרישות
- יש לעדכן Decisions כאשר מדובר בהחלטה מהותית
- יש לעדכן Context
- יש לעדכן TODO
- יש להסיר או לסמן מידע שהתיישן

---

## 15. פורמט פלט

כל טקסט המיועד לשימוש מחוץ לשיחה יוצג בפורמט מוכן להעתקה.

הדבר כולל:

- Prompts
- קוד
- Markdown
- SQL
- JSON
- YAML
- Commit Messages
- Pull Request Descriptions
- מסמכים
- תיעוד
- הוראות

אין לערבב הסברים בתוך הטקסט המיועד להעתקה.

---

## 16. תקשורת עם המשתמש

התקשורת מתבצעת בעברית.

קוד, Commands ומונחים מקצועיים יכולים להישאר באנגלית.

כאשר נדרשת שאלה:

- היא תהיה רב-ברירתית
- יהיו מעט אפשרויות
- תסומן האפשרות המומלצת כאשר קיימת
- תתווסף אפשרות `אחר — טקסט חופשי`

אין לשאול שאלה שניתן לפתור באמצעות החלטה טכנית סבירה.
