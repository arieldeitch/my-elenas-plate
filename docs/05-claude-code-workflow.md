# Claude Code Workflow

## 1. מטרת המסמך

מסמך זה מגדיר כיצד Claude Code צריך לעבוד על פרויקט Nutrition Tracker באופן עצמאי, עקבי ובטוח.

המטרה היא לאפשר לכל סשן חדש:

- להבין במהירות את מצב הפרויקט
- לזהות את המשימה הבאה
- לבצע שינויים ללא תלות בזיכרון שיחה
- לשמור על ה-Scope והארכיטקטורה
- לעדכן קוד ותיעוד יחד
- להשאיר נקודת המשך ברורה לסשן הבא

## 3. בדיקת מצב ראשונית

לפני שינוי קוד יש לבדוק:

- Branch פעיל
- `git status`
- Commits אחרונים
- קבצים לא שמורים
- Build קיים
- Type Check
- Tests קיימים
- Environment variables נדרשים
- מצב Supabase migrations
- האם קיימת סטייה בין הקוד למסמכים

אין לדרוס שינויים מקומיים שאינם מובנים.

אם קיימים שינויים מקומיים:

1. לזהות מי יצר אותם ומה מטרתם ככל שניתן.
2. לשמר אותם.
3. להשתלב בהם או להפריד את העבודה בבטחה.
4. לא לבצע Reset, Checkout או Clean הרסני ללא הצדקה מפורשת.

---

## 4. תכנון משימה

לפני כתיבת קוד יש להגדיר בקצרה:

- מטרת המשימה
- דרישות רלוונטיות
- קבצים או שכבות צפויים להשתנות
- Acceptance Criteria
- סיכונים אפשריים
- בדיקות נדרשות

אין להפוך משימה קטנה לתכנון ארכיטקטוני רחב.

כאשר דרישה ברורה, יש להמשיך עצמאית ללא בקשת אישור מיותרת.

---

## 5. סדר ביצוע מועדף

לכל משימה יש לפעול בסדר הבא:

1. להבין את הזרימה הקיימת.
2. לזהות Root Cause או Gap.
3. לבחור פתרון מינימלי ועקבי.
4. לעדכן Schema או Migration אם נדרש.
5. לעדכן Data Access ו-Domain Logic.
6. לעדכן UI.
7. להוסיף או לעדכן Validation.
8. להוסיף בדיקות לזרימות קריטיות.
9. להריץ Quality Gate.
10. לעדכן תיעוד.
11. לבצע Commit ממוקד.
12. לבצע Push כאשר הדבר חלק מהמשימה או מזרימת הסיום.

---

## 6. עבודה עם Lovable

Lovable משמש בעיקר עבור:

- יצירת UI/UX ראשוני
- בניית מבנה מסכים
- יצירת רכיבים בסיסיים
- האצת POC

לאחר סנכרון ל-GitHub:

- GitHub הוא מקור הקוד המרכזי.
- Claude Code רשאי לשפר, לתקן ולארגן את הקוד.
- אין לבנות מחדש את כל המוצר רק משום שהקוד נוצר ב-Lovable.
- יש לשמור ככל האפשר על Design Language שכבר אושר.
- שינוי רחב בעיצוב דורש דרישה מפורשת או בעיית UX מהותית.

---

## 7. עבודה עם Supabase

לפני שינוי Database יש לבדוק:

- migrations קיימות
- generated types
- RLS policies
- foreign keys
- constraints
- seed data
- השפעה על נתונים קיימים

כל שינוי Database חייב:

- להיות בקובץ Migration
- להיות ניתן לביקורת ב-Git
- לכלול RLS כאשר נדרש
- לכלול Indexes רלוונטיים
- להימנע משינוי ידני שאינו מתועד
- לעדכן Types לאחר השינוי, כאשר הפרויקט משתמש ב-generated types

אין להשתמש ב-Service Role בצד ה-Client.

## 8. עבודה עם נתונים קיימים

אין למחוק, לאפס או להחליף נתונים קיימים כדי להקל על הפיתוח.

כאשר נדרש שינוי Schema:

- יש להעדיף Migration לא הרסנית
- יש להוסיף Backfill כאשר נדרש
- יש לתמוך זמנית בערכים ישנים כאשר הדבר בטוח
- יש לתעד כל מגבלת Migration
- יש להכין Rollback או Remediation כאשר הסיכון משמעותי

## 9. בדיקות חובה

לכל שינוי יש להריץ את הבדיקות המתאימות מתוך הכלים הקיימים בפרויקט.

Quality Gate מינימלי:

- Build
- Type Check
- Lint
- Tests רלוונטיים
- בדיקת imports ו-dead code
- בדיקת Mobile בסיסית
- בדיקת Loading, Empty ו-Error states
- בדיקת שמירה וסנכרון כאשר רלוונטי

לשינויים בנתונים או חישובים יש להוסיף בדיקות לוגיות ממוקדות.

אין לדווח שמשימה הושלמה כאשר בדיקה מהותית נכשלה.

---

## 10. טיפול בשגיאות

כאשר פקודה או בדיקה נכשלת:

1. לקרוא את הפלט המלא הרלוונטי.
2. לזהות את השגיאה הראשונה המשמעותית.
3. לבדוק אם היא נוצרה מהשינוי הנוכחי.
4. לתקן Root Cause ולא רק סימפטום.
5. להריץ מחדש את הבדיקה הרלוונטית.
6. לבדוק שלא נוצרה Regression.

אין להסתיר שגיאות באמצעות:

- ביטול Type Checking
- שימוש נרחב ב-`any`
- השבתת Lint rules ללא הצדקה
- מחיקת Tests
- Catch ריק
- התעלמות מ-Promise rejection

## 13. Commit ו-Push

Commit צריך להיות:

- ממוקד
- בעל Message ברור
- ללא Secrets
- ללא קבצי Build זמניים
- כולל תיעוד רלוונטי
- לאחר Quality Gate סביר

פורמט מומלץ:

`type(scope): concise description`

דוגמאות:

- `feat(meals): add skipped meal status`
- `fix(sync): prevent duplicate offline retries`
- `docs(context): update project handover`
- `test(weight): cover delta calculation`

יש לבצע Push כאשר המשתמש ביקש זאת, כאשר מדובר בנקודת עצירה, או כאשר זהו נוהל העבודה הפעיל בפרויקט.

---

## 14. דו"ח סיום סשן

בסיום סשן יש לדווח:

- מה בוצע
- אילו קבצים מרכזיים השתנו
- אילו בדיקות עברו
- אילו בדיקות לא בוצעו ומדוע
- Commit hash ו-Branch, אם בוצע Commit
- האם בוצע Push
- מה נותר פתוח
- מה הצעד הראשון המומלץ לסשן הבא

הדו"ח צריך להיות עובדתי ולא שיווקי.
